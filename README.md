# Sugarscape Cultivation Simulator

一个用 Go 编写的大规模修仙世界模拟器。项目以 Sugarscape/agent-based simulation 的思路为基础，在二维世界中模拟凡人、修士、灵气、宗门、突破、移动、战斗和死亡等系统之间的长期反馈。

当前项目更像一个可调参的模拟实验场，而不是游戏。你可以观察诸如“有限灵气是否会抑制修士规模”“宗门是否会在高阶修士中占据更高比例”“资源竞争会如何影响死亡率和突破率”这类问题。

## Features

- 1000 x 1000 默认世界地图，支持百万格级别灵气与凡人人口模拟
- 凡人系统：出生、死亡、部落分布、凡人转化为修士
- 灵气系统：全图灵气分布、灵泉、灵脉、洞天福地、恢复、消耗、死亡返还和永久损耗
- 修炼系统：练气、筑基、金丹、元婴、化神五个境界
- 突破系统：灵气阈值、连续维持时间、失败冷却、金丹冲击元婴失败死亡
- 战斗系统：同格触发、战力判断、攻击欲望、资源竞争、战斗损耗和死亡返还
- 移动系统：基于当前灵气、灵气上限潜力、回复速度、探测范围和追击目标的移动决策
- 宗门系统：高灵气聚集地动态立宗、宗门影响范围、宗门战力、同宗助势、高阶导师突破加成
- 控制台实时摘要：境界分布、天地灵气、宗门统计、宗门/散修占比、各境界攻击性、追踪修士行为、死亡原因
- CSV 导出：默认写入 `output/stats.csv`

## Requirements

- Go 1.25+

项目目前没有外部依赖。

## Quick Start

```bash
go run .
```

默认运行参数：

- 世界大小：`1000 x 1000`
- 随机种子：`42`
- 并行 worker：`8`
- 每年 tick 数：`10`
- 默认执行：`300000` tick
- 每 `10s` 打印一次摘要
- 每 `5min` 自动暂停，按任意键继续，按 `q` 退出
- 可用 `Ctrl+C` 中途退出，退出前会正常输出摘要并导出 CSV

Web dashboard 模式：

```bash
go run . -web -port 18080
```

- 启动后每运行 `1min` 导出一组分析数据到 `output/analysis/`
- 运行满 `5min` 自动暂停

运行测试：

```bash
go test ./...
```

静态检查：

```bash
go vet ./...
```

## Output

控制台会周期性输出类似信息：

- 总修士数量、凡人数量、各境界人数
- 金丹及以上修士的灵气平均值和最大值
- 天地灵气总量、10/90 分位、低灵气/高灵气格子的灵气占比
- 宗门战力表：
  - 人数
  - 最高战力
  - 战力值，按百万显示为 `战力值(M)`
  - 练气、筑基、金丹、元婴、化神人数
- 宗门/散修在每个境界的人数和占比
- 随机追踪 4 位修士的本段行为
- 各境界死亡原因汇总
- 元婴、化神诞生事件

结束时会导出：

```text
output/stats.csv
```

CSV 中包含 tick/year、修士数量、凡人总数、平均灵气、平均年龄、平均战力、平均攻击性、死亡、出生、突破、凡人转化等聚合数据。

Web dashboard 每分钟额外导出：

```text
output/analysis/analysis_001m_tick_00000000.json
output/analysis/analysis_001m_tick_00000000.csv
```

JSON 是 dashboard 当前完整快照，CSV 是截至该分钟的累计统计快照。

## Core Model

### Realms

默认境界包括：

| 境界 | 灵气倍率 | 战力倍率 | 基础寿元 | 基础突破概率 |
| --- | ---: | ---: | ---: | ---: |
| 练气 | 2 | 1 | 120 | 10% |
| 筑基 | 6 | 3 | 250 | 5% |
| 金丹 | 20 | 10 | 500 | 3% |
| 元婴 | 60 | 30 | 1000 | 2% |
| 化神 | 200 | 100 | 3000 | 0% |

个体实际寿元会在基础寿元的 60% 到 100% 之间波动。

### Breakthrough

修士突破需要：

- 灵气达到上限的 `95%`
- 连续维持一段时间：
  - 练气 -> 筑基：50 tick
  - 筑基 -> 金丹：200 tick
  - 金丹 -> 元婴：500 tick
  - 元婴 -> 化神：1000 tick
- 不处于突破冷却期

散修突破概率会被压低，宗门修士有额外优势：

- 散修突破概率为基础概率的 65%
- 宗门基础突破概率提高 30%
- 宗门演化出的风格会进一步修正突破概率
- 同宗门高一阶修士越多，导师加成越高
- 导师加成使用饱和函数，避免无限滚雪球

### Sects

世界初始化时没有预设宗门。散修在高灵气区域持续聚集，且该区域发生足够战斗死亡后，才会演化出宗门。立宗还需要候选区内至少有 1 位元婴及以上，或 4 位金丹。立宗门槛较高，宗门之间也有较大的排他半径，避免地图上快速刷出过多宗门。

宗门战力值定义为：

```text
sum(combat_power^2)
```

宗门成立后会记录宗址、影响半径、立宗 tick 和触发战死数；影响半径内的散修会被纳入宗门，后续在宗址附近转化的新练气修士也会加入最近宗门。

宗门有扩张欲望，但扩张不是无条件扩大半径。系统会在既有据点附近寻找灵泉或灵脉级资源点，并计算资源收益、成员维护成本、已有据点成本、距离成本和与其他宗门争夺的成本。净收益为正时才会建立新据点，扩张成本会由宗门成员分摊消耗当前灵气。

## Project Structure

```text
.
├── engine/                    # 通用 agent-based simulation 引擎
│   ├── agent.go
│   ├── grid.go
│   ├── stats.go
│   └── world.go
├── scenarios/cultivation/      # 修仙世界场景实现
│   ├── config.go               # 核心参数
│   ├── scenario.go             # 初始化世界和注册系统
│   ├── mortal.go               # 凡人、转化、新生修士
│   ├── sect.go                 # 动态宗门形成、宗址和入宗范围
│   ├── cultivation.go          # 修炼和突破
│   ├── movement.go             # 移动和追击
│   ├── interaction.go          # 同格交互和攻击欲望
│   ├── combat.go               # 战斗结算
│   ├── lifecycle.go            # 寿元、低灵气死亡、日常消耗
│   └── environment.go          # 灵气恢复
├── STRATEGY.md                 # 当前策略规则的详细说明
├── main.go                     # CLI 入口和控制台输出
└── output/                     # 运行后生成 CSV
```

## Configuration

主要参数分布在：

- `engine/config.go`
- `scenarios/cultivation/config.go`
- `main.go`

当前还没有 CLI flags 或配置文件。如果要调参，可以直接改这些默认值。后续比较适合抽成 TOML/YAML 配置，方便批量实验。
